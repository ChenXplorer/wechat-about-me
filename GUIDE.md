# About-Me from WeChat — Claude Code 执行指南

> **如果你是 Claude Code，用户给你 @ 了这个文档**：用户希望你帮 ta 用自己的微信聊天记录产出一份《About Me》Markdown 长文档（行为画像 / 自我理解 / 对外输出素材）。按本文档从上到下执行。
>
> **关于风格定制**：本 GUIDE 的默认风格是"严肃报告 + 反占星/反标签"。如果用户给你的版本里 Layer 0 硬规则（第 4 节）或章节结构（第 3.4）有改动（例如允许 MBTI 4 字母、加入星座对照、改成 DISC 测评风、或换成鸡汤风），以用户版本为准——这是设计上允许的定制点。其他设计（两层架构、高保真度加权、Skill 三层辐射、分批处理）建议保留，它们是通用的方法论骨架。

## 0. 入口分支：用户有没有聊天记录 txt？

**先问用户一句**："你的微信聊天记录 txt 已经导出了吗？如果有，告诉我所在目录；如果没有，我带你导出。"

### 分支 A：用户有 txt

问清楚：
- txt 所在目录绝对路径
- 导出工具（weflow / memotrace / 其他），格式可能不同需要先抽样看一下
- 时间跨度（若用户不确定，脚本跑完阶段 0 会自动得出）

然后跳到第 1 节。

### 分支 B：用户还没导出 — 引导 weflow 导出

告诉用户以下步骤（weflow 是当前主流的开源微信记录导出工具之一）：

1. **环境要求**：weflow 在 Windows 上运行最稳。Mac 因为近几版 macOS 的系统安全限制，直接从本机提取微信数据库常常会被 AMFI 拦住，走 Windows VM 更顺。如果用户只有 Mac：
   - 可以用 Parallels / UTM 装 Windows 11 VM
   - 把手机上的聊天记录通过"聊天记录迁移与备份"迁移到 Windows 微信
   - 在 Windows 上运行 weflow

2. **用 weflow 导出**：
   - 从 weflow 官方仓库（GitHub 上搜 `weflow wechat`）下载最新 release
   - 按工具提示解锁数据库密钥（需要微信处于登录状态）
   - 导出格式选 **txt（纯文本）**，不是 html/csv
   - 导出范围：**所有联系人 + 所有群聊，时间跨度越长越好**
   - 导出位置：建议单独放一个文件夹（比如 `~/Desktop/wechat-export/texts/`）

3. **导出完成后回来告诉你**：用户把 txt 所在目录路径发给你，然后继续下一步。

### 如果用户用的不是 weflow

只要导出格式是"时间戳 + 发送者 + 消息内容"结构的 txt 文件（每个对话一个文件），都能用。格式差异在阶段 0 的解析脚本里处理即可。

## 1. 前置告知（开始执行前必须让用户知情）

在跑任何脚本前，明确告诉用户：

1. **数据会经过 Claude**：虽然 61 万级的全量消息不会出本机（本地 Python 处理），但经过规则筛选后的"高信号"子集（一般是全量的 1-3%）会随你的对话发给 Claude 做语义分析。这些样本**会包含 ta 最私密的情感、决策、家庭对话**。
2. **Claude 的默认策略**：数据 30 天保留期、不用于训练（以 Anthropic 官方条款为准）。
3. **让用户确认是否继续**。接受后才开始；不接受就停下——用更弱的本地模型做这事难产出可用质量。

## 2. 整体架构

```
┌──────────────────────────────────────────────────────────┐
│ Layer 1 — 本地 Python（全量消息在这层处理，不出本机）   │
│   解析 → 统计 → 规则粗筛 → 高信号 jsonl                 │
└────────────────────────┬─────────────────────────────────┘
                         │ 筛后 ~1-3% 的"高信号"子集
                         ▼
┌──────────────────────────────────────────────────────────┐
│ Layer 2 — Claude（你）分批处理                           │
│   分类 → 聚类 → 归纳 → 章节草稿 → 自审 → 定稿           │
└────────────────────────┬─────────────────────────────────┘
                         ▼
                 final/about_me.md
```

**铁律**：绝不把全量消息直接发给 Layer 2。Layer 1 的职责是降噪 + 筛选；Layer 2 的职责是语义理解 + 归纳 + 写作。

## 3. 六阶段执行清单

**每个阶段结束都必须做一次 checkpoint**：把产出的关键数字或抽样展示给用户，让 ta 确认"数字是否符合直觉、有没有反常像 bug、要不要调整规则"。用户说继续才进下一阶段。**不要连跑完所有阶段再给用户看**——中间 checkpoint 是这个流程最关键的质量保障机制。

### 阶段 0：解析归一化（本地 Python）

目标：把 `texts/*.txt` 解析成结构化 jsonl。

每条消息字段：
```
{
  "chat_id", "chat_type" ("private"|"group"), "chat_name",
  "ts", "sender", "is_me", "text", "msg_type"
}
```

`msg_type` 需要识别：text / image / sticker / voice / file / link / redpacket / transfer / location / miniapp / card / voip / system。

**⚠️ 阶段 0 最容易踩的坑**：**微信导出在群聊中不会用"我"标记自己**，而是用你在群里的真实昵称。解析完后必须做这个 checkpoint：
1. 列出所有群聊里 Top 发送者名字
2. **主动问用户**："你在群聊里常用的昵称是哪些？（可能不止一个。确认后我会把这些昵称也标记为'is_me=True'）"
3. 拿到昵称后更新 `ME_ALIASES` 集合，重跑阶段 0

如果用户只想跳过群聊分析（业主群、工作群这类事务性聊天对人格画像价值低），承认这些群里 ta 的发言识别为零，私聊 + 兴趣群已足够支撑分析。

**checkpoint 报告**：总消息数、私聊/群聊数、时间跨度、你发出/接收比例、消息类型分布。让用户判断是否合理。

### 阶段 1：统计骨架 + HTML 仪表盘（本地 Python）

产出单个 `dashboard.html`（用户浏览器打开）+ 细分 JSON + PNG 图。

必须包含：
- 月度消息量趋势（总量 + 你发出 + 你发出的私聊 vs 群聊）
- 24 小时活跃分布（按年叠加，看作息漂移）
- Top 50 私聊对象：消息量 / 互动天数 / 你发起占比 / 中位响应时长
- 你的消息长度分布（按年，box plot）
- 高频词 / emoji / 微信表情 Top N
- TF-IDF 话题种子 Top 200

**⚠️ 阶段 1 的坑**：词频分析前必须剥掉 `[XXX]` 格式的微信表情占位符（`[破涕为笑]` `[旺柴]` 等），否则这些表情名会被分词成"高频词"污染结果。正则：`re.sub(r'\[([^\[\]]{1,8})\]', '', text)`，然后再做 jieba 分词。

**checkpoint**：让用户浏览器打开 dashboard，给 ta 一个简要 checklist：
- 月度趋势看起来合理吗？有没有异常为 0 的月份（多半是解析 bug）？
- Top 高频词里有没有明显不应该上榜的？
- Top 私聊对象符合直觉吗？响应时间分布符合直觉吗？

用户反馈没 bug 后继续。

### 阶段 2：信号粗筛（本地 Python）

按规则从全量消息里筛 7 类高信号样本。每条记录必须附带：前 3 条上下文 + 后 3 条上下文 + `fidelity_score` + `trigger`（说明为什么被命中）。

**fidelity_score 规则（0.0-1.0）**：
- 基础分 0.5
- 深夜时段（22:00-03:00）× 1.3
- 1v1 私聊 × 1.3；群聊 × 0.5
- 连续对话 ≥ 10 轮 × 1.2
- 含决策/情绪词密度高 × 1.2
- 最终裁到 [0, 1.0]

**七类信号的筛选规则**：

| 文件 | 筛选思路 |
|---|---|
| `teaching.jsonl` | 用户发的 ≥ 100 字长消息且含解释性关键词（"因为/比如/具体来说/思路是/首先/1./"）OR 发 ≥ 200 字且对方下条非撤回/纯表情的实质回应 |
| `asked.jsonl` | 对方私聊用疑问句（含"怎么/为啥/能不能/请问/如何/什么时候"或问号结尾）且长度 10-200 字，排除寒暄黑名单（"在吗/吃了吗/睡了吗"等），且用户下一条有 ≥ 5 字实质回应 |
| `emotion.jsonl` | 用户发的 ≥ 6 字消息含情绪词（正负两个小词典各约 30-50 词）或正负向微信表情，同时记录 polarity |
| `decisions.jsonl` | 用户发的 ≥ 15 字消息含决策词（"所以/决定/算了/不如/果断/打算/犹豫/权衡"） |
| `avoidance.jsonl` | 用户连续 ≥ 3 条短回复（每条 < 10 字）后对话静默 ≥ 6h |
| `contradictions.jsonl` | 用户发的 8-300 字含主张词（"我觉得/我认为/一定/必须/永远/从不/绝对"） |
| `topic_seeds.json` | 复用阶段 1 的 TF-IDF Top 200 |

**⚠️ 阶段 2 的坑**：规则筛选必有 10-20% 假阳性（陈述句被当疑问句、"麻烦"被当负向情绪）。**不要在规则层追求完美——Layer 2 用语义再过滤效果好得多**。规则的目标是把全量降到可分析量级。

**checkpoint**：对每类信号抽 5-10 条高保真度样本给用户看，判断假阳性率是否 ≤ 20%。高于此则调规则重跑。

### 阶段 3：Claude 信号解读（分批处理，你自己跑）

读 signals 目录下的 jsonl，分批产出 12 个 analysis JSON：

| # | 输入 | 输出 |
|---|---|---|
| 1 | `teaching.jsonl` | `skill_taught.json` — 真掌握的领域 + 证据计数 + 每个领域的 pattern |
| 2 | `asked.jsonl` | `skill_perceived.json` — 他人眼中专长 + Top askers + 典型问题 |
| 3 | 上两个合并 | `skill_gap.json` — matches / teach-not-asked / asked-not-teach |
| 4 | `emotion.jsonl` + timeline | `emotional_arc.json` — 低谷 / 高光 / 触发器 |
| 5 | `decisions.jsonl` | `values.json` — 反复出现的决策原则 |
| 6 | `avoidance.jsonl` + 全维度 | `blind_spots.json` — 限制性模式 |
| 7 | `topic_seeds.json` + 统计 | `interests.json` — 按证据强度排序的真实兴趣 |
| 8 | messages 按对象类型采样 | `voice_shifts.json` — 对不同对象的语言面具 |
| 9 | 跨维度交叉 | `tag_scores.json` — 预设的 15-20 个标签候选打分，**只写行为描述不写裸标签名** |
| 10 | 决策+情感+语言样本 | `mbti_dims.json` — E/I、S/N、T/F、J/P 四维独立分析，**不出 4 字母结论** |
| 11 | `contradictions.jsonl` | `contradictions.json` — 价值观内部矛盾 |

**给每批的通用输出规则**：
- 证据样本 < 5 条的结论必须标注 `[样本不足]`，不硬填
- 每条结论带 `evidence_count` 和样本平均保真度
- 不写抽象标签名（"纠结体""秒回选手"），写可观察行为规则（"在场景 X 下反复出现行为 Y，频次 Z"）

**checkpoint**：每批处理完给用户一个 JSON 概要，让 ta 过目后继续。

### 阶段 4：章节草稿

8 章结构（可按用户偏好微调）：

```
序章    数据来源、能说什么不能说什么（边界声明）
第 1 章  时间轴：按年拆解生活主旋律
第 2 章  社交引力图：核心圈 / 中间层 / 外围
第 3 章  语言指纹 & 行为画像（含 MBTI 4 维独立证据）
第 4 章  情绪与心理模式
第 5 章  兴趣真实分布
第 6 章  ⭐ Skill 三层辐射图（核心）
第 7 章  ⭐ 盲区与成长建议（核心）
第 8 章  价值观与决策模式
终章    给 N 年后的你一封信
附录    ## 自我修正记录（留给用户读后追加）
```

**效率**：8 章之间相对独立，可以用 Claude Code 的 subagent（Agent 工具）并行写，最后合并。

**⚠️ 每章 prompt 头部必须嵌入的 Layer 0 写作硬规则**（见第 4 节，不可违背）。

### 阶段 5：自审 + 定稿

合并所有章节 → `final/about_me.md` → 做两轮检查：

**第一轮：正则扫描**（脚本自动）
- MBTI 4 字母：`\b[EI][SN][TF][JP]\b`
- 星座：白羊|金牛|双子|巨蟹|狮子|处女|天秤|天蝎|射手|摩羯|水瓶|双鱼
- 裸标签名（话痨|纠结体|秒回选手|计划狂|浪漫主义）
- 人生导师腔（记住|你值得|相信自己|加油）
- 正能量封顶（可爱之处|这恰恰说明）
- 心理诊断标签（回避型依恋|高敏感人群|ADHD|焦虑症|抑郁）
- `[样本不足]` 标注数量（至少几处才算诚实）

**第二轮：交叉一致性**（你自己读）
- 章节之间的结论有没有矛盾？（如果有，要么修正，要么在文档里明确指出"这两点存在张力"）
- 每个优点后面有没有阴影面？
- 盲区章节有没有被软化？

末尾追加 `## 自我修正记录` 空节，给用户读后填修正意见。

**交付完成后告诉用户**：文档路径 + 打开方式 + 如何填修正记录以便下轮迭代（ta 填完后再 @ 文档给你，你产出 v2）。

## 4. Layer 0 写作硬规则（每章 prompt 头部必须包含）

```
你在写一份严肃的行为分析报告（不是朋友圈长文，不是 DISC 测评）。
风格：严肃报告 + 适度温度（像高管教练写的分析，不像心理鸡汤）。

硬规则（违反任何一条都必须重写）：
1. 不引用聊天原文——用抽象归纳 + 统计语言
2. 不做心理学诊断（不写"回避型依恋""高敏感人群""ADHD"等标签）
3. 不输出 MBTI 4 字母结论（只写 4 维各自行为证据）
4. 不使用星座
5. 不输出裸人格标签名（话痨/纠结体/秒回选手等），翻译为"在 X 场景下反复出现 Y 行为，频次 Z"
6. 证据样本 < 5 条必须标 [样本不足]，不靠推断硬填
7. 每个优点配一个阴影面；每个缺点配可操作改善动作
8. 盲区章节不软化结论，不加"但这也是你的可爱之处"
9. 不做人生导师腔，不写"记住，你值得被爱"式结尾
```

## 5. 设计原则背后的"为什么"（遇到边界情况时判断依据）

### 为什么不引用聊天原文

三个原因：
1. **隐私**：文档可能给别人看、几年后再翻——聊天原文里永远有不应该被二次传播的内容
2. **避免停留在表层**：读者看到"你说过 XXX"会停在具体句子层面，错过模式
3. **强迫归纳**：把观察抽象成"频次 + 场景 + 行为描述"，这才是真正有洞察的层级

遇到用户要求引用原文时，向用户解释这个原则并坚持。除非 ta 有明确的、超过隐私代价的理由。

### 为什么 MBTI 4 字母不输出

4 字母结论没信息量，只有占星效果。但 4 个维度**独立用行为证据反推**是有价值的——可以出现"偏 I 但熟人场景看起来像 E"这种有语境的判断，拒绝 INTJ / ENFP 这类标签。

### 为什么每个优点配阴影面

没有纯粹的优点。"秒回"的阴影是"几乎不留思考缓冲"；"信息密集决策"的阴影是"决策拖延到 deadline"。强制配阴影面能让文档不变成讨好文案。

### 为什么盲区不软化

LLM 天然倾向给盲区补一句"但这也是你的可爱之处"。必须明确禁止。盲区就是盲区——软化等于让读者忽略它，也就让报告失去最有价值的部分。

### 为什么高保真度加权

深夜消息 / 1v1 私聊 / 长对话后段 / 含决策情绪密度高——这些语境下的话更接近真实想法，少些社交表演。fidelity_score 让你在 context 有限时把最值得被看到的数据排在前面。

### 为什么 Skill 三层辐射图

识别能力的三个独立维度：
1. **你教过什么** = 真掌握
2. **别人问你什么** = 他人眼里的专长
3. **词汇熟练度** = 不经意领域

关键洞察往往在错位处——教了但没人问（隐藏能力）、被问了但没系统教（最容易对外变现的方向）。

## 6. 常见坑清单（按踩坑概率排序）

1. **群聊里"我"不是"我"**：导出工具用用户真实昵称，阶段 0 必须处理
2. **微信表情被当高频词**：分词前剥掉 `[XXX]` 占位符
3. **同一个人多个昵称**：微信允许同一人在不同群用不同名字；重要人物需要建立昵称映射
4. **误把家人昵称当同事**：LLM 不知道用户家人叫什么。**任何被识别为"关键人物"的陌生名字都要在 checkpoint 让用户确认身份**。例："楠子"可能是配偶昵称，不是同事
5. **规则误判**：asked 会把陈述句当问句、emotion 会把"麻烦"当负向——不用死磕，让语义层过滤
6. **LLM 取悦用户**：必须严格执行 Layer 0 硬规则，否则章节会变成鸡汤
7. **过度拟合周期**：数据点少时不要归纳"X 个月周期"。举例：如果用户只换过两三次工作，别说"18 个月跳槽一次"——这是过拟合
8. **低估早期数据稀疏**：头一两年的数据量往往只有后几年的一小部分，章节里要声明
9. **把短回复后沉默当心理回避**：大多数时候只是对话自然结束，不是防御
10. **忘记做 checkpoint**：每个阶段停下给用户看产出。这不是浪费时间——是唯一能保证整个流程不偏方向的机制

## 7. 需要向用户主动问询的时刻清单

在执行过程中，以下场景必须**主动问用户，不要自己替 ta 决定**：

- 阶段 0 后：ta 在群聊里的昵称有哪些？
- 阶段 0 后：是否要分析所有群聊，还是只分析私聊 + 兴趣群（跳过业主群/工作群）？
- 阶段 1 后：dashboard 里有没有看起来像 bug 的数据？
- 阶段 2 后：信号抽样质量 OK 吗？要不要调规则？
- 阶段 3 每批后：JSON 概要过目
- 遇到"关键人物身份识别"时：这个名字是谁？（配偶/同事/朋友/家人）
- 阶段 5 交付后：告诉 ta 如何填自我修正记录

## 8. 关于分享和多轮迭代

### 用户想分享部分内容时
- 从章节中抽取可对外段落，**脱敏后**放单独文件夹（如 `sanitized_excerpts/`）
- 脱敏规则：去掉具体人名、公司名、数字金额，保留行为模式描述
- 最终输出前提醒用户亲自过一遍，检查有没有无意识暴露的内容

### 用户想给配偶/家人看时
- 涉及对方的分析部分**请用户先征得对方同意**再分享
- 最安全做法：给家人的版本只保留关于用户自己的章节（如第 3、4、6、7、8 章）

### 下轮迭代（v2）
用户读完后在 `## 自我修正记录` 里追加修正意见时，会再 @ 这个文档或 about_me.md 给你。那时按 ta 的反馈：
1. 定位涉及错误事实的章节段落
2. 在原地修正（不是整体重写）
3. 在"自我修正记录"节记录这轮变更
4. 保持其他正确的部分不动

## 9. 最后一句

这份流程的真正价值不在脚本也不在 LLM——在于每个阶段的 **checkpoint 较真**和 **Layer 0 硬规则的严格执行**。遇到用户说"差不多就行"时：checkpoint 可以合并但不能跳过；硬规则可以商量表达方式但不能放弃约束。
